# 升學大師 v2.0 - 重新定位執行計畫書

**專案名稱：** 升學大師 - 高職升學路徑發現引擎  
**計畫版本：** v2.0 重新定位版  
**計畫日期：** 2026-05-10  
**計畫期間：** 2026-05-10 ~ 2026-07-31 (12週)  
**負責團隊：** 產品、設計、技術、營運

## 📋 執行摘要

### 🎯 專案目標
將升學大師從「學習歷程管理平台」重新定位為「高職升學路徑發現引擎」，解決 99% 高職生不知道自己有 6 種升學管道的核心痛點。

### 💡 核心策略
**從補習班思維 → 發現者思維**
- **不是：** 記錄成績、展示成就、管理歷程（補習班在做的事）
- **而是：** 幫助高職生發現不知道的升學路徑、轉化經歷為資產、以終為始規劃

### 🎭 差異化定位
**核心角色定位：** 高職生 = 升學領域的「探索者」→「拓荒者」→「建築師」→「啟動者」→「開創者」

## 🚨 問題陳述與市場機會

### 核心痛點分析
```
痛點 1：資訊不對稱 (嚴重度：★★★★★)
├─ 99% 高職生不知道自己有 6 種升學管道
├─ 只知道統測分發，以為這是唯一路徑
└─ 結果：8% 高職生不必要地跨考學測

痛點 2：能力看不見 (嚴重度：★★★★)
├─ 做了很多事，但不知道對升學有什麼價值
├─ 社團活動、專題製作、競賽參與的升學價值被忽略
└─ 結果：感覺「讀高職沒用」，失去動力

痛點 3：結構性劣勢 (嚴重度：★★★★)
├─ 外語群、商管群：無技術士證照可考
├─ 以为高職路走不通，被迫跨考學測
└─ 結果：放棄專業興趣，選擇不適合的路徑

痛點 4：家長看不見 (嚴重度：★★★)
├─ 家長只看到成績，看不到孩子在成長
├─ 不知道如何陪伴，只會施壓
└─ 結果：親子關係緊張，失去支持系統
```

### 市場規模與機會
```
目標市場：台灣高中職學生
├─ 總人數：約 40 萬人
├─ 每年升學需求：約 13 萬人（3 年級合計）
├─ 願意付費家長：預估 20%（約 2.6 萬人）
└─ 市場規模：NT$ 2.4B/年（每人年均 NT$ 9,000）

未被滿足的需求：
├─ 116 選考策略需求（2027 年改革）
├─ 外語群、商管群的特殊指導需求
├─ 「能力轉化」的具體操作指引
└─ 家長看見孩子成長的迫切需求
```

## 🎯 重新定位策略

### 產品定位重新定義

#### v1.0 定位（問題）：
```
升學大師 = 學習歷程管理平台
核心功能：記錄成就、展示成果、AI 分析
目標用戶：所有高中生
差異化：AI 驅動 + 家長系統
問題：同質化嚴重，補習班思維
```

#### v2.0 定位（目標）：
```
升學大師 = 高職升學路徑發現引擎
核心功能：發現未知路徑 + 轉化經歷為資產 + 以終為始規劃
目標用戶：高職生（專注）+ 家長（付費）
差異化：多角色成長系統 + 群體專屬策略
價值主張：「你不必跟隨人群，可以開創屬於自己的路徑」
```

### 核心差異化優勢

| 維度 | 補習班/競品 | 升學大師 v2.0 |
|------|-------------|---------------|
| **思維** | 成績導向 | 發現導向 |
| **功能** | 記錄、展示、管理 | 發現、轉化、規劃 |
| **用戶狀態** | 被動接收者 | 主動探索者 |
| **價值** | 提高分數、錄取更好 | 找到更適合的路徑 |
| **時機** | 考試前衝刺 | 早期規劃、持續成長 |
| **情感連結** | 焦慮、壓力 | 探索、驚喜、成就感 |

## 🎭 多角色系統設計

### 五大角色定義

#### 🔍 探索者 (Explorer)
```
目標：發現升學世界中的未知路徑和隱藏機會
定義：剛踏入升學世界的探索者，好奇心旺盛，願意了解不同選項
階段：註冊前/註冊初期
占比：95% (初期用戶絕大多數)
核心任務：了解 6 種升學管道、認識群體機會、打破認知盲點
成功標準：「我原來有這麼多選擇！」的驚喜感
```

#### 🗺️ 拓荒者 (Pathfinder)
```
目標：在未知領域開創屬於自己的路徑
定義：勇于開路，願意探索他人未走之路，找到最適合自己的管道
階段：完成基礎探索後的進階用戶
占比：80% (中期活躍用戶)
核心任務：深入了解管道細節、記錄第一個經歷、建立基礎策略
成功標準：「我發現了一條適合我的路徑」的確定感
```

#### 🏗️ 建築師 (Architect)
```
目標：系統化設計升學藍圖和策略
定義：具備全局視野，善於結構化思維，重視效率和策略
階段：有明確目標和一定經歷的中級用戶
占比：60% (深入參與用戶)
核心任務：設計升學藍圖、規劃時間表、分析能力缺口
成功標準：「我知道該做什麼，按部就班執行」的控制感
```

#### ⚡ 啟動者 (Catalyst)
```
目標：將日常活動轉化為升學資產，執行建築師設計的藍圖
定義：重實際行動，能在日常活動中發現價值，持續調整優化
階段：積極參與的高級用戶
占比：40% (核心付費用戶)
核心任務：記錄轉化、執行藍圖、AI 分析、家長互動、社群交流
成功標準：「每個活動都有明確目的和價值」的成就感
```

#### 🏆 開創者 (Trailblazer)
```
目標：成功達成升學目標，成為他人的引路人
定義：已經走通路徑的經驗者，願意分享和指導新人
階段：已達成目標的專家用戶
占比：15% (KOL 和意見領袖)
核心任務：分享故事、指導學弟妹、建立影響力、社羣影響
成功標準：「我幫助了其他人找到路徑」的影響力
```

### 角色進化路徑
```
探索者 (95%) → 拓荒者 (80%) → 建築師 (60%) → 啟動者 (40%) → 開創者 (15%)
    ↓            ↓            ↓            ↓           ↓
  好奇      →  發現    →   規劃    →   行動    →   影響

轉換門檻：
探索者 → 拓荒者：完成 6 種管道了解 + 註冊 + 閱讀群體機會
拓荒者 → 建築師：記錄 1+ 經歷 + 設定目標 + 感受驚喜
建築師 → 啟動者：完成藍圖設計 + 執行 5+ 行動 + 收集 10+ 資產
啟動者 → 開創者：達成目標 + 分享故事 + 指導 5+ 新人
```

## 🏗️ 技術架構重構

### 資料庫架構擴展

#### 新增用戶角色管理表
```sql
-- 用戶角色追蹤表
CREATE TABLE user_roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id) NOT NULL,
  current_role VARCHAR(20) NOT NULL, -- explorer, pathfinder, architect, catalyst, trailblazer
  level SMALLINT DEFAULT 1, -- 1-15
  experience_points INTEGER DEFAULT 0,
  secondary_roles TEXT[], -- ["mentor", "community_leader"]
  group_affiliation VARCHAR(20), -- 商管群/外語群/工科群等
  joined_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(user_id)
);

-- 角色升級歷史表
CREATE TABLE role_evolution (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id) NOT NULL,
  from_role VARCHAR(20),
  to_role VARCHAR(20) NOT NULL,
  upgraded_at TIMESTAMPTZ DEFAULT now(),
  reason TEXT, -- 升級原因
  milestone_achieved TEXT -- 達成的里程碑
);

-- 角色成就系統
CREATE TABLE role_achievements (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id) NOT NULL,
  achievement_type VARCHAR(50) NOT NULL, -- discovery_first, path_created, blueprint_complete, first_execution, etc.
  achievement_data JSONB,
  unlocked_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_user_roles_current ON user_roles(current_role, level);
CREATE INDEX idx_role_evolution_user ON role_evolution(user_id, upgraded_at);
```

## ⚠️ 風險評估與緩解策略

### 風險 1: 角色系統過度複雜
```
風險：5 個角色可能讓用戶困惑
影響：高
緩解策略：
├─ 強化「自動升級」機制，減少用戶選擇負擔
├─ 設計「智能推薦」預設角色
├─ 提供「快速模式」，跳過複雜角色系統
```

### 風險 2: 用戶不願意重新定位
```
風險：現有用戶可能習慣 v1.0 定位
影響：中
緩解策略：
├─ 漸進式導入，逐步引入新概念
├─ 保留 v1.0 功能，但加強 v2.0 特色
├─ 說明「升級」的額外價值，而非「替換」
```

### 風險 3: 技術實現複雜度
```
風險：多角色系統開發和維護成本高
影響：高
緩解策略：
├─ 採用模組化設計，角色系統可插拔
├─ 優先實施核心角色，次要角色後續
├─ 建立完善的角色轉換自動化
└─ 代碼重用最大化，避免重複開發
```

### 風險 4: 錯誤的「角色分層」感覺
```
風險：高級角色看不起低級角色
影響：低
緩解策略：
├─ 強調「每個角色都有重要性」
├─ 設計「角色互補」機制，如 Trailblazer 需要回饋 Explorer
├─ 社群活動設計，鼓勵跨角色交流
└─ 避免在稱號或語調上產生「等級差」
```

### 風險 5: 現體名稱混淆風險
```
風險：用戶不知道自己該扮演什麼角色
影響：中
緩解策略：
├─ 根據用戶行為自動推薦角色
├─ 提供清晰的「角色說明書」
├─ 在關鍵節點引導角色轉換
└─ 保持角色定義的簡潔明確
```

```typescript
// 角色狀態管理器
class RoleStateMachine {
  private user: User
  
  getCurrentRole(): UserRole {
    const role = this.user.current_role
    const level = this.user.level
    return { ...ROLE_DEFINITIONS[role], level }
  }
  
  canUpgradeTo(): UserRole[] {
    const currentRole = this.getCurrentRole()
    const xp = this.user.experience_points
    
    // 檢查升級條件
    return Object.keys(ROLE_DEFINITIONS)
      .filter(role => this.meetsRequirements(role, xp))
      .filter(role => this.isValidUpgrade(role))
  }
  
  upgradeTo(role: UserRole): void {
    // 角色升級邏輯
    this.createRoleEvolutionRecord(role)
    this.updateCurrentRole(role)
    this.grantRoleCapabilities(role)
    this.sendCongratulationMessage(role)
  }
  
  meetsRequirements(role: UserRole, xp: number): boolean {
    return ROLE_REQUIREMENTS[role].every(req => 
      this.user[req.field] >= req.value
    )
  }
}

const ROLE_REQUIREMENTS = {

  ],
  catalyst: [

  ]
}
```

```typescript
// 角色權限系統
class RoleCapabilityManager {
  getAvailableActions(user: User): AvailableAction[] {
    const role = this.getCurrentRole(user)
    return ROLE_CAPABILITIES[role]
  }
  
  canAccessFeature(user: User, feature: string): boolean {
    const role = this.getCurrentRole(user)
    return FEATURE_PERMISSIONS[role].includes(feature)
  }
  
  grantCapabilities(user: User, role: UserRole): void {
    const caps = ROLE_CAPABILITIES[role]
    // 解鎖對應功能的使用權
```
